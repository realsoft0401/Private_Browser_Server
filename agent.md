# Agent Notes

当前 `Private_Browser_Server` 只做第一阶段骨架与 bind 主线，不回搬 old 的完整业务域。

## 文档收口规则

- 同一项新能力、同一组接口或同一条治理链路，默认只保留两份文档：
  - 一份正式文档
  - 一份草案文档
- 不要把同一能力继续拆成很多平行 md，例如再单独拆“字段字典”“OpenAPI 草案”“SSE 草案”三四份并列文件。
- 如果确实需要字段字典、OpenAPI 草案、SSE 结构、数据库映射，这些内容应优先合并进同一份草案文档，而不是继续新增第三、第四份平行文档。
- 正式文档负责收口：
  - 业务语义
  - 职责边界
  - 数据库关系
  - 状态变化
  - 成功失败口径
- 草案文档负责收口：
  - OpenAPI path / method
  - 请求响应结构
  - 字段字典
  - SSE 事件草案
  - 示例 payload
- 如果发现已经把同一能力拆成太多 md，应优先做文档合并和入口清理，再继续扩展内容。
